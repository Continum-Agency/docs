## Configuration File

The `.continum.json` file controls how the CLI scans your project. It's created automatically when you run `continum init`.

```json .continum.json
{
  "scanOnCommit": true,
  "sandbox": "employee_confidential",
  "block": ["CRITICAL", "HIGH"],
  "warn": ["MEDIUM"],
  "ignore": [
    ".env.example",
    "**/*.test.ts",
    "**/fixtures/**",
    "**/mocks/**"
  ],
  "patterns": {
    "custom": [
      "ACME-[0-9]{6}",
      "internal-project-[a-z]+"
    ]
  }
}
```

<Info>
  This file should be committed to your repository so all team members use the same configuration.
</Info>

<Warning>
  **Security:** The `.continum.json` file does NOT contain your API credentials. Your API key is stored securely in `~/.continum/credentials.json` (never committed) and is created when you run `continum login`.
</Warning>

## Configuration Options

### scanOnCommit

Enable or disable pre-commit scanning.

```json
{
  "scanOnCommit": true
}
```

**Values:**
- `true` - Scan files before every commit (default)
- `false` - Disable automatic scanning

<Warning>
  Disabling this removes protection from your commits. Only do this temporarily for debugging.
</Warning>

---

### sandbox

The Continum sandbox to use for background auditing.

```json
{
  "sandbox": "employee_confidential"
}
```

**Value:** String - Your sandbox slug

This determines where audit results are sent for AI analysis. You can create sandboxes in your dashboard.

---

### block

Risk levels that should block commits.

```json
{
  "block": ["CRITICAL", "HIGH"]
}
```

**Values:** Array of risk levels
- `CRITICAL` - Definite credentials (API keys, passwords)
- `HIGH` - Likely credentials (high-entropy strings in suspicious contexts)
- `MEDIUM` - Possible credentials (moderate confidence)
- `LOW` - Low-confidence detections

**Recommended:** `["CRITICAL", "HIGH"]`

---

### warn

Risk levels that show warnings but allow commits.

```json
{
  "warn": ["MEDIUM"]
}
```

**Values:** Array of risk levels (same as `block`)

Warnings are displayed but don't prevent commits. Useful for gradual rollout or low-confidence patterns.

---

### ignore

Files and patterns to skip during scanning.

```json
{
  "ignore": [
    ".env.example",
    "**/*.test.ts",
    "**/fixtures/**",
    "**/mocks/**",
    "docs/**"
  ]
}
```

**Values:** Array of glob patterns

Common patterns to ignore:
- `.env.example` - Example environment files
- `**/*.test.ts` - Test files
- `**/fixtures/**` - Test fixtures
- `**/mocks/**` - Mock data
- `**/*.md` - Documentation files

<Tip>
  Use glob patterns for flexible matching. `**` matches any directory depth.
</Tip>

---

### patterns.custom

Custom regex patterns to detect.

```json
{
  "patterns": {
    "custom": [
      "ACME-[0-9]{6}",
      "internal-project-[a-z]+",
      "prod_key_[a-zA-Z0-9]{32}"
    ]
  }
}
```

**Values:** Array of regex patterns

Add company-specific credential formats that aren't in the built-in patterns.

## Example Configurations

### Strict Mode

Block all violations, no warnings:

```json
{
  "scanOnCommit": true,
  "sandbox": "production",
  "block": ["CRITICAL", "HIGH", "MEDIUM", "LOW"],
  "warn": [],
  "ignore": [
    ".env.example"
  ]
}
```

### Gradual Rollout

Warn on medium/low, block only critical/high:

```json
{
  "scanOnCommit": true,
  "sandbox": "development",
  "block": ["CRITICAL", "HIGH"],
  "warn": ["MEDIUM", "LOW"],
  "ignore": [
    ".env.example",
    "**/*.test.ts",
    "**/fixtures/**"
  ]
}
```

### Development Environment

More lenient for development:

```json
{
  "scanOnCommit": true,
  "sandbox": "dev_sandbox",
  "block": ["CRITICAL"],
  "warn": ["HIGH", "MEDIUM"],
  "ignore": [
    ".env.example",
    "**/*.test.ts",
    "**/fixtures/**",
    "**/mocks/**",
    "docs/**",
    "examples/**"
  ]
}
```

## Updating Configuration

After changing `.continum.json`, the changes take effect immediately on the next commit. No need to restart or reinitialize.

```bash
# Edit configuration
vim .continum.json

# Test with a commit
git add .
git commit -m "test new config"
```

## File Locations and Security

Understanding where Continum stores different types of data:

### `.continum.json` (Project Root)

**Location:** Your project's root directory  
**Purpose:** Scanner configuration  
**Contains:**
- Sandbox name
- Block/warn rules
- Ignore patterns
- Custom patterns

**Security:** ✅ Safe to commit to git

### `~/.continum/credentials.json` (Home Directory)

**Location:** Your home directory (`~/.continum/`)  
**Purpose:** Authentication credentials  
**Contains:**
- API URL
- API key

**Security:** ❌ NEVER commit this file - it's in your home directory, not your project

### `~/.continum/patterns.json` (Home Directory)

**Location:** Your home directory (`~/.continum/`)  
**Purpose:** Cached patterns from Continum API  
**Contains:**
- Downloaded pattern library
- Team-approved patterns

**Security:** ✅ Safe (no credentials), but no need to commit

## Troubleshooting

<AccordionGroup>
  <Accordion title="Configuration not found">
    Run `continum init` to create the configuration file.
  </Accordion>
  
  <Accordion title="Changes not taking effect">
    Ensure `.continum.json` is in your repository root, not a subdirectory.
  </Accordion>
  
  <Accordion title="Too many false positives">
    Add specific files or patterns to the `ignore` list, or move violations to `warn` instead of `block`.
  </Accordion>
  
  <Accordion title="Invalid JSON">
    Validate your JSON syntax. Use a JSON validator or your editor's JSON support.
  </Accordion>
</AccordionGroup>
